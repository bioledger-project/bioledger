Structured metadata, not spreadsheets

Free-text CSV columns are ambiguous — an AI has to guess what "organism" or "sample" means. ISA-Tab gives BioLedger structured, ontology-backed metadata instead: sources, samples, characteristics, and assays with formal relationships between them. That structure is what lets an LLM reason reliably about your data — which files belong to which sample, which organism they came from, what assay produced them — instead of pattern-matching on column names.

Run anywhere you have Docker or Conda

BioLedger tool specs are language- and execution-agnostic: a container image plus a declarative command template, input/output contract, and parameters. Today BioLedger runs tools locally via Docker — you're not tied to a third-party platform or orchestrator to get your analysis done. That means every run is fully observable: BioLedger records the exact command, container, and file hashes, giving you reproducibility and traceability without depending on someone else's infrastructure. Conda support is planned as an alternative runtime for environments where Docker isn't available (e.g. shared HPC clusters). Cluster environments using Singularity aren't supported yet.

Think in experiments, not workflows

Researchers don't design a pipeline up front — they load some data, try a tool, look at the output, and decide what's next. BioLedger is built around that reality: every action you take (tool runs, imports, LLM calls, notes) is recorded in an append-only ledger as you work. The workflow and RO-Crate aren't something you author — they're crystallized after the fact from the ledger, as an artifact that makes what you did reproducible for someone else (including future you).

LLM where it earns its keep

The LLM is used only where judgment genuinely helps — suggesting a next step, resolving how one tool's output should feed the next tool's input, fixing malformed imports. Everything else is deterministic and programmatic: execution, file hashing, provenance linking, and artifact generation. Because the ledger records every action and every LLM message as a first-class, file-based artifact, the whole history is independently verifiable — you don't have to trust a model's summary of what happened.

Roadmap

Near-term: Galaxy API key support (run tools against a Galaxy instance you already have credentials for), native Nextflow workflow execution, and Conda as an alternative local runtime to Docker for HPC and shared-cluster environments.

Execution expansion: We want BioLedger to run on the platforms researchers already use — submitting batch jobs to HPC clusters, supporting Singularity for sites without Docker, and cloud execution. Today it's local Docker only; these are the next frontiers.

Longer-term: have BioLedger generate its own library assets — ingesting Galaxy tools or nf-core modules as BioLedger tool specs, or turning an NCBI BioProject accession into an ISA-Tab study by querying NCBI directly. Until that's automated, these assets are produced by hand and contributed to the library repos or kept locally.

Explore the libraries

BioLedger hosts two small, growing public libraries of ready-to-use assets. They're searchable here, and discoverable directly from the CLI (bioledger library list, bioledger study list) — tools are auto-imported on first use.

Tool Spec Library

Curated bioinformatics tool specs (samtools, bwa-mem2, GATK, bcftools, and more) ready to run with bioledger tool import.

Study Library

Reference genomes and experimental datasets as ISA-Tab studies, downloadable with bioledger study load.
